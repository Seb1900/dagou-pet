# Dagou 商业化与自动更新实施计划

- 负责人：Seb1900
- 目标平台：Windows 10/11 x64
- 当前版本：0.2.0
- 文档状态：规划中
- 自动更新优先级：官网直发版 P0

## 目标

建立可审计、可签名、可停止发布并可前滚修复的 Windows 商业发行链路，为
NSIS 安装版提供可靠的应用内更新。Portable 版本只检查并引导下载，Steam
版本由 Steam 管理更新。

## 许可前提

Seb1900 是当前项目的版权方。PolyForm Noncommercial 1.0.0 约束获得该许可的
第三方，不限制 Seb1900 商业发行自己拥有版权的代码与素材。

计划采用双重许可：

- 公开源码继续使用 PolyForm Noncommercial，限制第三方未经授权的商业使用。
- 商业二进制使用单独的最终用户许可协议（EULA）。
- 图片、图标、音频、参考代码和未来外部贡献仍需分别留存权属或商业授权证明。
- 未来接收外部代码贡献前，应采用贡献者许可协议（CLA）或等价的书面授权。

## 分发方式

| 分发方式 | 检查 | 下载 | 安装 | 应用内表现 |
| --- | --- | --- | --- | --- |
| NSIS 官网版 | `electron-updater` | 应用内 | 重启覆盖安装 | 完整更新界面 |
| Portable | 可检查 | 浏览器或明确下载 | 用户手动替换 | 明示“便携版” |
| Steam | Steam | SteamPipe | Steam | 禁用自带更新器 |
| 开发/解包版 | 禁用 | 禁用 | 禁用 | 显示开发版本 |

不同分发方式必须使用明确的构建标识，不能只依靠用户设置判断。Portable 不承诺
可靠的原地更新，Steam 版本不得同时访问独立更新源。

## 阶段 0：权属与发行身份（P0）

### 工作项

- 确定最终发行主体：Seb1900 本人或其控制的公司。
- 确定 Windows 签名证书中的 Publisher 名称。
- 在首次商业公开发行前冻结以下身份：
  - `appId`
  - 产品名、安装目录名和可执行文件名
  - 用户数据目录
  - 官网、隐私政策和支持地址
- 建立 `assets/ASSET_PROVENANCE.md`，逐项登记所有精灵图、图标和 WAV：
  - 文件路径与 SHA-256
  - 作者或录音者
  - 创作或取得日期
  - 原始工程、录音母带或来源
  - 是否包含第三方内容
  - 商业使用依据与修改记录
- 对 Mikutap、Dagou-Tap-New 等参考项目标明仅借鉴思路还是实际复用；任何代码、
  音频或图像复用都必须单独核对许可证。
- 建立 `THIRD_PARTY_NOTICES.md`，收录实际随应用分发的第三方组件。
- 为代码仓库启用双因素认证、受保护分支和正式发布环境审批。

### 验收标准

- 全部代码与素材均归 Seb1900 所有，或具有可归档的商业授权。
- LICENSE、NOTICE、README 和商业 EULA 的适用对象没有冲突。
- 发行身份和 `appId` 已形成书面决定，不再随意变更。

## 阶段 1：签名发布基础（P0）

### 工作项

- 取得与最终发行主体一致的 Windows 代码签名能力。
- 优先使用云签名、HSM 或 OIDC 临时授权；私钥不得进入仓库或普通 CI Secret。
- 所有签名使用可信时间戳，并建立证书轮换和吊销流程。
- 在 `electron-builder.yml` 中固定 Publisher 和签名策略。
- CI 对安装包、主程序和卸载程序执行签名验证。
- 每个版本生成并归档：
  - NSIS 安装包
  - Portable 包
  - `latest.yml`
  - blockmap
  - `SHA256SUMS.txt`
  - 第三方许可清单
  - SBOM

### 验收标准

- 全新 Windows 10/11 环境可以安装、启动和卸载。
- 默认路径与自定义安装路径均通过测试。
- SignTool 或 `Get-AuthenticodeSignature` 显示有效签名与时间戳。
- 安装包内主程序和 `uiohook-napi` 原生模块均可正常加载。
- CI 日志与产物不包含私钥或长期发布凭据。

## 阶段 2：Windows 发布流水线（P0）

### 工作项

- 新增 `.github/workflows/release-windows.yml`。
- 只允许受保护的 `vX.Y.Z` Tag 触发正式发布。
- 校验 Tag、`package.json` 版本和发布说明一致。
- 固定 Node.js 版本，并依次执行：
  1. `npm ci`
  2. `npm run verify`
  3. 构建与打包
  4. 打包冒烟测试
  5. 代码签名
  6. 签名、哈希与产物验证
  7. 生成候选发布
- 第三方 GitHub Actions 固定到完整 commit SHA。
- 正式发布使用独立环境和人工批准。
- 先上传安装包和 blockmap，确认完整后最后上传更新清单。
- 同一版本号的二进制不可覆盖或重新上传。
- 面向中国用户时，自有 HTTPS 对象存储/CDN 作为主更新源，GitHub Releases
  作为镜像或候选产物存储。

### 验收标准

- 一个受保护 Tag 能生成完整且已签名的候选发布。
- 任一测试、签名或上传步骤失败时，生产更新清单保持不变。
- 普通 PR 和非受保护分支无法发布正式版本。
- CDN 中二进制使用不可变长缓存，更新清单使用短缓存或 `no-cache`。

## 阶段 3：应用内更新（P0）

### 工作项

- 将固定版本的 `electron-updater` 加入生产依赖。
- 新建主进程更新服务，Renderer 不直接访问更新网络。
- 通过受限且校验 sender 的 IPC 提供：
  - 获取更新状态
  - 手动检查
  - 开始下载
  - 取消下载
  - 失败重试
  - 重启安装
  - 稍后退出时安装
- 状态机至少覆盖：
  - `idle`
  - `checking`
  - `up-to-date`
  - `available`
  - `downloading`
  - `downloaded`
  - `canceled`
  - `error`
- 设置页展示当前版本、最新版本、更新说明、文件大小、进度、速度和错误分类。
- 托盘增加“检查更新”，启动约 10 秒后静默检查一次；无更新时不打扰。
- 设置 `autoDownload=false`，由用户确认后下载。
- 只在 `app.isPackaged` 且属于 NSIS 官网版时启用完整更新。
- 对离线、超时、403/404、磁盘不足、签名失败和损坏文件提供可操作提示。
- 更新说明使用纯文本或严格清理后的内容渲染。
- 本地诊断日志不得记录按键、活动窗口、进程名或输入节奏。

### 安全要求

- 更新源只能是代码中受信任的 HTTPS 域名。
- 更新包必须同时通过 `latest.yml` 中的 SHA-512 和 Windows 代码签名验证。
- 远端错误栈、任意 HTML 和任意 URL 不直接交给 Renderer。
- 外部下载链接由主进程校验协议和域名后再打开。
- 开发版和解包版默认不访问生产更新源。

### 验收标准

使用两个真实签名版本完成 A 到 B 的完整测试：

- 安装 A，修改音量、缩放、位置、按键与演奏模式。
- A 能检测到 B，并正确展示版本和说明。
- 下载进度、速度、取消与重试均有效。
- 重启安装后版本变为 B，原有设置全部保留。
- 更新前全局键盘 Hook 和音频资源正常释放。
- 无更新、离线、慢网、磁盘不足、损坏包和签名错误均完成测试。
- Windows 10/11、默认路径和自定义安装路径均通过。

## 阶段 4：隐私、设置迁移与基础信任（P0）

### 工作项

- 新增正式 `PRIVACY.md`，并在设置页提供入口。
- 明确全局键盘 Hook 仅在本地处理，不保存或上传：
  - 输入内容与键码历史
  - 组合键历史
  - 窗口标题与进程名
  - 输入节奏或打字统计
- 披露更新请求会向更新服务暴露 IP、应用版本和常规 HTTP 请求信息。
- 为设置文件增加 schema 版本和显式迁移函数。
- 保留现有原子写入，并增加 `.bak`、损坏恢复和一次性恢复提示。
- 设置页增加当前版本、更新通道、发行主体、许可、隐私和反馈入口。
- 开机启动默认关闭，必须由用户主动开启且可随时撤销。

### 验收标准

- 至少两个历史设置样本可以升级且不丢失数据。
- 损坏 `settings.json` 后可以恢复，并向用户显示一次性提示。
- 应用和发布页均可访问版本、隐私、许可与反馈信息。
- 隐私声明与程序真实行为一致。

## 遥测与崩溃上报（P1）

Sentry 和产品遥测不是首版商业发行的强制 P0。引入前必须满足：

- 默认关闭或取得明确同意。
- 使用严格的事件字段白名单。
- 禁止采集按键、扫描码、窗口标题、进程名、用户名、路径和音频。
- 禁用与输入有关的 breadcrumbs。
- 对崩溃栈中的路径和标识符脱敏。
- 说明保存期限、删除方式、服务区域和第三方处理方。
- 用户可在设置中撤回同意。

更新检查不应宣称“完全匿名”：即使没有产品遥测，服务端仍可能看到 IP。

## 后续能力（P1/P2）

### P1

- 稳定版与测试版更新通道。
- 5% -> 25% -> 100% 分批发布。
- 内容包统一清单、预览、启用、禁用、卸载与版本兼容检查。
- 中英文、多显示器验证、设置导入导出。
- 用户主动触发的短视频或 GIF 录制与分享。

### P2

- Steam 版本与 SteamPipe 发布。
- DLC 或支持者版本权益验证。
- 付费声音包、皮肤包和主题组合包。
- 用户自制内容、审核、作者授权与分成。

## 回滚与事故处理

- 永不覆盖已经发布的安装包、清单和 blockmap。
- 发现问题后立即停止扩量，并阻止尚未升级的用户继续取得问题版本。
- 已安装问题版本的用户不依赖自动降级；以更高版本号发布前滚修复。
- 至少保留最近三个稳定版本的已签名安装包、清单与 blockmap。
- 危险的设置迁移前创建备份，并保持至少一个版本的向后可读性。
- 证书泄露时停止发布、吊销证书、轮换凭据并发布新的签名修复版。
- 每次事故记录影响版本、处置过程、恢复时间和防复发措施。
